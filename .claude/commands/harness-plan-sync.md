---
description: docs/plan.md에서 다음 미완료 Phase를 탐색하여 task.md + checklist.md를 자동 생성
argument-hint: Optional — 특정 Phase 이름 지정 시 해당 Phase로 고정 (예: "Phase 5")
---

`docs/plan.md`를 분석하여 다음 미완료 Phase의 상세 실행 계획을 자동으로 생성합니다.

## 실행 흐름 (반드시 아래 순서대로)

### 0. 사전 검증 (history.md 갱신 확인)

실행 전 `docs/history.md` 최근 수정 상태를 확인합니다.

**중단 조건 — 아래 중 하나라도 해당하면 task.md를 덮어쓰기 전에 중단합니다:**

1. **history.md 미갱신:** 이번 세션에서 docs/history.md가 갱신되지 않은 경우
   - 안내: "⚠️ history.md에 완료 내용을 먼저 기록하세요. task.md를 덮어쓰기 전 완료 내용이 보존되어야 합니다."
   - 사용자에게 `/harness-docs-update` 실행을 권고하고 대기

2. **plan.md 전체 완료 감지:** plan.md의 모든 Phase에 `✅ 완료` 또는 `[x]` 처리가 완료된 경우
   - 안내: "✅ 현재 sprint 전체 완료. docs/history.md에 Sprint 완료 기록 후 새 plan.md를 작성하세요."
   - 새 plan.md 작성 전까지 task.md 교체 보류

**계속 조건:** history.md가 이번 세션 중 갱신되었거나, 사용자가 명시적으로 계속 진행을 요청한 경우

---

### 1. docs/plan.md 전체 읽기

`docs/plan.md` 파일을 읽어 전체 Phase 목록을 파악합니다.

### 2. 대상 Phase 결정

**기본 (argument 없을 때):** 아래 조건을 모두 만족하는 Phase 중 **첫 번째**를 선택합니다.
- `✅ 완료` 텍스트가 Phase 제목에 없음
- `[ ]` 체크박스 항목이 1개 이상 존재

**argument 지정 시:** `$ARGUMENTS`에서 Phase 이름을 읽어 해당 Phase를 고정으로 선택합니다.
(예: "Phase 5" 입력 → Phase 5의 [ ] 항목을 대상으로 함)

### 3. 대상 Phase의 [ ] 항목 전체 추출

선택된 Phase에서 `[ ]` 체크박스가 붙은 항목을 전부 목록화합니다.

### 4. 각 항목별 상세 분석

`[ ]` 항목마다 아래 4가지를 분석합니다:

```
① 산출물 명세
   - 파일명 및 경로
   - 역할 (한 문장)

② 구현 단계 (3~7단계 번호 목록)
   - 구체적인 실행 순서
   - 각 단계에서 수행할 작업 명시

③ 수정/생성 대상 파일 목록
   - 신규 생성: 파일 경로 + 역할
   - 수정: 파일 경로 + 수정 내용 요약

④ 검증 방법
   - 파일 존재 확인 명령어 또는
   - 특정 내용 grep 명령어 또는
   - 기능 동작 확인 기준
```

### 4.5. 병렬 세션 분석

Step 4에서 분석된 각 항목의 수정 대상 파일/디렉토리를 기반으로 병렬 실행 가능 여부를 판단합니다.

**단일 세션 판정 (아래 중 하나라도 해당하면 병렬 불필요):**
- `[ ]` 항목이 3개 이하
- 수정 파일이 모두 같은 디렉토리에 속함
- 전체 항목에 순차 의존성 존재 ("A 완료 후", "B 착수 조건" 등)

→ 단일 세션 판정 시 Step 5 템플릿의 "세션 전략"을 간략 출력하고 넘어갑니다.

**병렬 분석 (항목 4개 이상 + 수정 범위 분산):**

```
① 격리 경계 자동 감지:
   - base/[stack]/     → 스택별 독립
   - .claude/skills/[name]/  → 스킬별 독립
   - .claude/agents/[name].md → 에이전트별 독립
   - .claude/hooks/[name].*   → 훅별 독립 (단, settings.json 등록은 공유)
   - src/[module]/     → 모듈별 독립 (실 프로젝트)

② 공유 파일 감지 (일괄 세션 전용):
   - docs/ 4문서 (plan.md, task.md, history.md, checklist.md)
   - settings.json, skill-rules.json
   - README.md 류 (agents/README, skills/README)
   - CLAUDE.md, base_code_plan.md, CHANGELOG.md

③ 의존성 체인 감지:
   - 항목 간 "선행 조건" 텍스트 존재 여부
   - "A 완료 후", "B 착수 조건" 등 패턴

④ 세션 그룹핑:
   - 독립 세션: 수정 범위 겹침 없음 → 병렬 가능
   - 순차 세션: 의존성 있음 → 선행 완료 후 실행
   - 일괄 세션: 공유 파일 수정 → 모든 병렬 세션 완료 후 마지막
```

---

### 5. docs/task.md 덮어쓰기

아래 표준 형식으로 `docs/task.md`를 완전히 새로 작성합니다:

```markdown
# Task — 현재 작업 상세 계획서

> **현재 Phase:** [Phase 이름] — [Phase 설명]
> **선행 완료:** [직전 Phase 이름] ([날짜]) — [직전 Phase 완료 내용]

---

## [Phase 이름] — [Phase 설명]

### 개요

| 항목 | 내용 |
|---|---|
| **목표** | [이 Phase의 최종 목표] |
| **산출물** | [생성/수정 파일 수 및 유형] |
| **의존성** | [선행 필요 작업 또는 "없음"] |

### 세션 전략

**[Step 4.5 결과에 따라 아래 두 가지 중 하나를 출력]**

**(A) 단일 세션 판정 시:**

```markdown
> 단일 세션 권장 — 항목 수 [N]개, 수정 범위 동일 디렉토리
```

**(B) 병렬 세션 분석 결과가 있을 시:**

```markdown
> 아래 세션 배정은 수정 범위 격리 분석 결과입니다.
> 같은 색 세션은 **동시 실행 가능**, 일괄 세션은 모든 병렬 세션 완료 후 실행.

| 세션 | 담당 항목 | 수정 범위 | 비고 |
|---|---|---|---|
| 🟦 세션 A | [항목 ID] | `[경로]` | 독립 |
| 🟩 세션 B | [항목 ID] | `[경로]` | 독립 |
| 🟧 일괄 | [항목 ID] | `settings.json`, `README.md` | A,B 완료 후 |

**격리 규칙:**
- 각 세션은 담당 수정 범위만 수정
- 공유 파일은 일괄 세션에서만 수정
- 항목이 3개 이하이면 단일 세션 권장 (병렬 오버헤드 > 이득)
```

### 6. docs/checklist.md 업데이트

`docs/checklist.md` 파일을 읽고, 파일 최상단에 새 Phase 체크리스트를 **추가(append가 아닌 prepend)**합니다.

**추가 형식:**

```markdown
## [Phase 이름] — 체크리스트

> **현재 Phase:** [Phase 이름] — [Phase 설명]
> **선행 완료:** [직전 Phase] — [직전 완료 내용]

---

### [단계 ID]: [단계 제목]

**파일 생성/수정:**
- [ ] [파일 경로] 존재 확인
- [ ] [핵심 구성 요소] 포함 확인

**내용 검증:**
- [ ] [검증 기준 1]
- [ ] [검증 기준 2]

---

(각 단계 반복)

*Phase [이름] 체크리스트 추가 — [날짜]*
```

실제로는 기존 checklist.md 내용 앞에 삽입하거나 현재 Phase 섹션 바로 위에 추가합니다.

### 7. 완료 안내 출력

```
✅ [Phase 이름] task.md + checklist.md 생성 완료

생성된 내용:
  → docs/task.md  — [단계 수]개 단계 상세 계획
  → docs/checklist.md — [단계 수]개 단계 체크리스트 추가

다음 작업: task.md의 첫 번째 [ ] 항목부터 순서대로 구현을 시작하세요.
```

---

## 추가 컨텍스트: $ARGUMENTS

argument가 비어있으면 plan.md에서 첫 번째 미완료 Phase를 자동 선택합니다.
argument에 "Phase 5"처럼 Phase 이름을 지정하면 해당 Phase를 고정으로 사용합니다.
