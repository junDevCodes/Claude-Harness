---

구현 완료 후 `/quality-gate` 판정 → NEEDS WORK 시 미충족 항목 수정 → 재판정 사이클을 **최대 3회** 반복합니다. NEXUS 패턴의 핵심 반복 개선 로직을 하네스 품질 체계에 이식한 커맨드입니다.

> **목적:** 수동 반복 검증 없이 DoD 충족까지 자동으로 루프. 3회 안에 PASS가 나오지 않으면 에스컬레이션하여 수동 검토를 요청합니다.

---

## 루프 흐름 다이어그램

```
[구현 완료 / 이미 구현된 경우]
         │
         ▼
  [Loop N/3] ──→ /quality-gate 판정
         │                │
         │    NEEDS WORK  │   PASS
         │ ←──────────────┤    │
         │   미충족 항목   │    ▼
         │   수정 실행     │  [완료 — PASS 출력]
         │   (N < 3)       │
         │                │
         │  N = 3 도달     │
         └────────────────→ [에스컬레이션 — 수동 검토 요청]
```

---

## 실행 흐름 (반드시 아래 순서대로)

### Step 1: 현재 작업 범위 및 DoD 파악

1. `docs/task.md`를 읽어 현재 Phase와 작업 대상을 파악합니다.
2. `docs/checklist.md`를 읽어 현재 Phase의 DoD 항목 전체를 확인합니다.
3. **argument 지정 시:** `$ARGUMENTS`에서 작업 대상 설명을 읽어 컨텍스트로 활용.
4. 루프 카운터를 **N = 0**으로 초기화합니다.

> 출력: `[작업 대상] DoD 항목 확인 완료. Dev-QA Loop 시작.`

---

### Step 2: 구현 / 수정 실행

**이미 구현이 완료된 경우:** 이 단계를 건너뛰고 Step 3으로 바로 이동합니다.

**구현이 필요한 경우:**
1. `docs/task.md`의 작업 항목을 순서대로 실행합니다.
2. 파일 생성, 코드 작성, 내용 추가 등 필요한 작업을 수행합니다.
3. 구현 완료 후 Step 3으로 이동합니다.

---

### Step 3: `/quality-gate` 판정 실행

루프 카운터를 **N = N + 1**로 증가시키고, 현재 Phase DoD 전항목 점검을 실행합니다.

**루프 상태 표시 (매 루프 시작 시 출력):**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔄 Dev-QA Loop [Loop N/3] — quality-gate 판정 시작
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**판정 실행:**
- `/quality-gate` 커맨드의 실행 흐름(Step 1~4)을 동일하게 수행합니다.
- 판정 결과(판정 표 + 총평)를 출력합니다.

**판정 결과에 따라:**
- **전체 PASS** → Step 5 (완료)로 이동
- **NEEDS WORK 존재** → Step 4 (수정)로 이동
- **N = 3이고 NEEDS WORK 존재** → Step 5 (에스컬레이션)로 이동

---

### Step 4: 미충족 항목 수정

**조건:** N < 3 이고 NEEDS WORK 항목이 존재할 때만 실행.

1. Step 3의 NEEDS WORK 항목 목록을 확인합니다.
2. **미충족 항목만** 선별하여 수정합니다 (이미 PASS인 항목 재수정 금지).
3. 수정 내용을 간략히 출력합니다:
   ```
   🔧 [Loop N/3] 수정 실행:
   - [미충족 항목 1]: [수정 내용 요약]
   - [미충족 항목 2]: [수정 내용 요약]
   ```
4. 수정 완료 후 Step 3으로 복귀하여 재판정합니다.

---

### Step 5: 결과 출력

**PASS 달성 시 (N ≤ 3):**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Dev-QA Loop 완료 — PASS [Loop N/3]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

루프 요약:
  Loop 1/3: [PASS / NEEDS WORK M개]
  Loop 2/3: [PASS / NEEDS WORK M개]  ← 해당하는 경우만
  Loop 3/3: [PASS]                   ← 해당하는 경우만

모든 DoD 항목 충족. 다음 단계로 진행 가능합니다.

권장 후속 작업:
  → /harness-docs-update — 4문서(history/checklist/task/plan) 업데이트
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**3회 초과 에스컬레이션 시 (N = 3, NEEDS WORK 잔존):**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️  Dev-QA Loop 에스컬레이션 — 3회 루프 후 미충족 항목 잔존
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

루프 요약:
  Loop 1/3: NEEDS WORK M개
  Loop 2/3: NEEDS WORK M개
  Loop 3/3: NEEDS WORK M개

자동 수정으로 해결되지 않은 항목 (수동 검토 필요):

1. [미충족 항목 1] — [미충족 이유 / 가능한 원인]
2. [미충족 항목 2] — [미충족 이유 / 가능한 원인]

권장 조치:
  1. 위 항목을 수동으로 검토하고 수정하세요.
  2. 수정 완료 후 /quality-gate를 직접 실행하여 재판정하세요.
  3. 필요 시 docs/checklist.md의 DoD 기준 재검토를 고려하세요.
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 연계 커맨드 / 에이전트

| 자산 | 연계 방식 |
|---|---|
| `/quality-gate` | Step 3에서 직접 호출 — DoD 판정의 실제 실행 엔진 |
| `/harness-docs-update` | PASS 달성 후 권장 — 4문서 일괄 업데이트 |
| `reality-checker` 에이전트 | 에스컬레이션 후 광범위 품질 판정이 필요한 경우 |
| `pipeline-orchestrator` 에이전트 | 복잡한 멀티 에이전트 작업의 전체 Dev-QA 사이클 조율 시 |

## 추가 컨텍스트: $ARGUMENTS

argument가 비어있으면 `docs/task.md`에서 현재 작업 대상을 자동으로 파악합니다.
argument에 작업 대상 설명(예: `"quality-gate.md 커맨드 파일"`)을 지정하면 해당 대상에 집중하여 루프를 실행합니다.
